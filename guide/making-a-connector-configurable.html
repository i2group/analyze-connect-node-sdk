    Making a connector configurable

    Depending on the nature of a connector, it might be necessary to allow administrators to configure some aspects of its behavior at the point of deployment. This configuration might include settings such as:

    • URLs, credentials, and other information for connecting to third-party services.
    • Controls over the behavior of a service, such as what data the service returns.

    As you develop your connectors, consider what configuration settings you might include to make them as flexible as possible for their deployers. Connectors can receive settings in a number of ways:

    • From a configuration file only
    • From an environment variable only
    • From a configuration file with optional environment variable override

    Connectors built with the i2 Connect Node SDK support configuration settings with any of the following types:

    • string
    • number
    • Boolean
    • string array
    • object

    Note: Any array- or object-based settings that you supply through environment variables must be specified as JSON strings. For example:

    • ["one", "two", "three"]
    • {"prop 1":"prop value 1", "prop 2":123, "prop 3":true}

    Providing settings in a configuration file

    To use file-based configuration settings, add them to the connectorSettings section of the top-level default.json file. The following example contains the definition of a setting named url:

    {
  "$schema": "https://i2group.github.io/analyze-connect-node-sdk/schemas/config-schema-2.0.json",
  ...
  "connectorSettings": {
    "url": "https://example-url.com"
  }
}

    If required, you can define settings in hierarchical sections to organize them. For example:

      "connectorSettings": {
    "url": "https://example-url.com",
    "query": {
      "timeout in seconds": 20,
      "allow wildcard searches": true,
      "document file search extensions": [".doc", ".txt", ".html"]
    }
  }

    Handling sensitive information

    Many connectors need to handle information that could pose a security risk if it were compromised, such as usernames and passwords. To safeguard such data:

    • Never store sensitive information in a physical file, including the top-level default.json file.

      This guidance is even more important if your source code is published to a publicly accessible Git repository.

    • Use environment variables to store sensitive information for your connector. You can then access the information through the i2 Connect API's settings namespace (see below), or directly by using process.env.

      To help with deployment, document any environment variables that you rely on in the README.md file that you distribute with your connector.

    Accessing configuration settings from service code

    Configuration settings are only useful if you can access them from within your service! The i2 Connect API provides functionality for doing so in the settings namespace (just import settings from @i2analyze/i2connect). Each function in the namespace can access a setting of a different type, and is named accordingly.

    For example, to read a file-based string value named url:

    import { settings } from '@i2analyze/i2connect';

const url = settings.getString({path: "url"});

    Alternatively, to read a string value from an environment variable named ENV_URL:

    import { settings } from '@i2analyze/i2connect';

const url = settings.getString({environmentVariable: "ENV_URL"});

    Or, to read a file-based string value named url that might be overridden by an environment variable named ENV_URL:

    import { settings } from '@i2analyze/i2connect';

const url = settings.getString({path: 'url', environmentVariable: "ENV_URL"});

    You can indicate that a configuration setting must be present by specifying the isRequired parameter when you attempt to fetch its value. For example:

    import { settings } from '@i2analyze/i2connect';

const url = settings.getString({path: "url"}, true});

    The call to settings.getString() then throws an error if the setting is not found.
